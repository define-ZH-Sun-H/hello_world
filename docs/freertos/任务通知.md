# FreeRTOS 任务通知使用指南

## 什么是任务通知

每个 FreeRTOS 任务内部都有一个 **32 位通知值** 和一个 **通知状态**。任务通知 API 利用这个内置的「信箱」实现 **极轻量级的 IPC**，不需要像队列或信号量那样额外创建内核对象。

```
┌──────────────┐     vTaskNotifyGiveFromISR()     ┌──────────────┐
│              │ ──────────────────────────────►  │              │
│  ISR / 任务  │                                   │  目标任务    │
│              │     xTaskNotify(ulValue)          │  通知值 +=1  │
│              │ ──────────────────────────────►  │              │
└──────────────┘                                   └──────────────┘
```

### 与队列、信号量的性能对比

| 同步方式 | 相对速度 | RAM 开销 | 适用场景 |
|---------|---------|---------|---------|
| **任务通知** | **1×**（最快） | 0（任务自带） | 点对点通信 |
| **二值信号量** | ~1.5× | 需要 `SemaphoreHandle_t` | 简单事件通知 |
| **队列** | ~2× | 需要 `QueueHandle_t` + 缓冲区 | ISR→任务数据传递 |
| **事件组** | ~1.5× | 需要 `EventGroupHandle_t` | 多条件等待 |

任务通知不仅更快，而且**省内存**——每个任务创建时已经预分配了通知值，无需额外创建内核对象。

### 任务通知的限制

| 特性 | 限制 |
|------|------|
| **通信方向** | 只能点到点（一个任务通知另一个任务） |
| **消费者数量** | 每次只有一个任务能消费通知值 |
| **数据量** | 只有 32 位值，不能传递结构体 |
| **广播** | ❌ 不支持 |

多生产者 / 多消费者场景 → 用队列或事件组。

## API 总览

```c
#include "freertos/FreeRTOS.h"
#include "freertos/task.h"

/* 模式一：轻量级信号量（Give / Take） */
BaseType_t xTaskNotifyGive(TaskHandle_t xTaskToNotify);            // 通知值 +1
void vTaskNotifyGiveFromISR(TaskHandle_t xTaskToNotify,
                            BaseType_t *pxHigherPriorityTaskWoken); // ISR 版（出参：是否需立即切换）
uint32_t ulTaskNotifyTake(BaseType_t xClearCountOnExit,
                          TickType_t xTicksToWait);                 // 等待并消费

/* 模式二：带值通知（Notify / Wait） */
BaseType_t xTaskNotify(TaskHandle_t xTaskToNotify,
                       uint32_t ulValue,
                       eNotifyAction eAction);
BaseType_t xTaskNotifyFromISR(TaskHandle_t xTaskToNotify,
                              uint32_t ulValue,
                              eNotifyAction eAction,
                              BaseType_t *pxHigherPriorityTaskWoken);
BaseType_t xTaskNotifyWait(uint32_t ulBitsToClearOnEntry,
                           uint32_t ulBitsToClearOnExit,
                           uint32_t *pulNotificationValue,
                           TickType_t xTicksToWait);

/* 查询 / 清除 */
uint32_t ulTaskNotifyValueClear(TaskHandle_t xTask,
                                uint32_t ulBitsToClear);
```

## 模式一：轻量级信号量（Give / Take）

等效于二值信号量，但不创建任何内核对象。ISR 通知任务「某件事发生了」，任务阻塞等待。

### 基本用法

```c
/* 全局句柄（目标任务保存自己的句柄供 ISR 使用） */
TaskHandle_t xKeyTaskHandle = NULL;

void key_task(void *pv)
{
    /* 保存自己的句柄 */
    xKeyTaskHandle = xTaskGetCurrentTaskHandle();

    while (1) {
        /* 阻塞等待通知（替代 xSemaphoreTake） */
        ulTaskNotifyTake(pdTRUE, portMAX_DELAY);

        /* 被唤醒后执行 */
        KeyInd_Scan();
        KeyInd_HandleEvents();
    }
}

void IRAM_ATTR key_isr_handler(void *arg)
{
    BaseType_t xHigherPriorityTaskWoken = pdFALSE;

    /* 发送通知（替代 xSemaphoreGiveFromISR） */
    vTaskNotifyGiveFromISR(xKeyTaskHandle, &xHigherPriorityTaskWoken);

    /* 如果 key_task 优先级更高，立即切换 */
    portYIELD_FROM_ISR(xHigherPriorityTaskWoken);
}
```

### `pxHigherPriorityTaskWoken` 参数说明

这是所有 `FromISR` 函数**共有的出参**，新手最容易忽略。

```c
BaseType_t xHigherPriorityTaskWoken = pdFALSE;   // ① 必须初始化为 pdFALSE

vTaskNotifyGiveFromISR(xKeyTaskHandle, &xHigherPriorityTaskWoken);  // ② 传入地址
portYIELD_FROM_ISR(xHigherPriorityTaskWoken);    // ③ 检查是否需要切换
```

**作用：** ISR 中调用通知函数后，内核会检查被唤醒的任务优先级是否高于当前被打断的任务。如果是，内核将 `xHigherPriorityTaskWoken` 设为 `pdTRUE`。

**为什么需要它：** ISR 返回时本应回到被打断的任务。但如果一个高优先级任务被唤醒了，我们想**立即执行它**而不是等下一个 tick 中断。三步走：

| 步骤 | 代码 | 含义 |
|------|------|------|
| ① 初始化 | `= pdFALSE` | 还没人需要被切换 |
| ② 传地址 | `&xHigherPriorityTaskWoken` | FromISR 函数会把结果写到这里 |
| ③ 切换 | `portYIELD_FROM_ISR(woken)` | 如果被设为 `pdTRUE`，ISR 返回时直接运行高优先级任务 |

**传 NULL 会怎样？**

```c
/* 偷懒写法：不传出参 */
vTaskNotifyGiveFromISR(xKeyTaskHandle, NULL);     // 通知照发，不检查优先级

// 后果：被唤醒的任务最多等 10ms（一个 tick）才能运行
// 100Hz 下，ISR→任务响应延迟从 0 变成 ~10ms
```

| 行为 | 传 `&变量` | 传 `NULL` |
|------|:---------:|:---------:|
| 通知发送 | ✅ 正常 | ✅ 正常 |
| 是否检查优先级 | ✅ 检查 | ❌ 不检查 |
| 被唤醒任务何时运行 | 立即（ISR 返回后） | 延迟到下一个 tick（最多 10ms） |
| 适用场景 | 高优先级任务需要及时响应 | 延迟不敏感的通知 |

### 消费端在哪？

这是最容易困惑的地方。理解一句话即可：

> **`ulTaskNotifyTake()` 消费的是调用者自己的通知值**，不需要传句柄。

```
生产者（ISR/任务）                                消费者（目标任务）
                                              
vTaskNotifyGiveFromISR(○, &w)              ulTaskNotifyTake(pdTRUE, portMAX_DELAY)
                       │                               ↑
                       └─── 告诉内核："给 ×× 任务      │
                            的通知值 +1"               └── 调用者就是自己
                            需要传目标句柄                  不需要传句柄
```

完整流程：

```
key_task 启动
    │
    ├── xKeyTaskHandle = xTaskGetCurrentTaskHandle()
    │   （保存自己的句柄，供 ISR 使用）
    │
    ├── ulTaskNotifyTake(pdTRUE, portMAX_DELAY)
    │   │
    │   └──► 阻塞等待 ──► ISR 触发 ──► vTaskNotifyGiveFromISR(xKeyTaskHandle, &w)
    │                              │           └── 通知 key_task 的通知值 +1
    │                              │
    │                              └──► ulTaskNotifyTake 返回（消费完成）
    │
    ├── 处理按键（KeyInd_Scan / KeyInd_HandleEvents）
    │
    └── 回到 ulTaskNotifyTake 继续阻塞等待
```

所以：
- **生产者（ISR）** → `vTaskNotifyGiveFromISR(xTaskHandle, &woken)` — **需要**目标任务句柄
- **消费者（目标任务）** → `ulTaskNotifyTake(pdTRUE, portMAX_DELAY)` — **不需要**句柄，消费自己的通知值

### `ulTaskNotifyTake()` 参数说明

```c
uint32_t ulTaskNotifyTake(
    BaseType_t xClearCountOnExit,  // pdTRUE=清0, pdFALSE=减1
    TickType_t xTicksToWait        // 超时
);
```

| `xClearCountOnExit` | 效果 | 等效对象 |
|--------------------|------|---------|
| `pdTRUE` | 通知值归零 | **二值信号量**（一次唤醒消费一个事件） |
| `pdFALSE` | 通知值减 1 | **计数信号量**（多次 Give 只需一次 Take 消费） |

### Give/Take 等价对照表

| 操作 | 信号量方式 | 任务通知方式 |
|------|----------|------------|
| 创建 | `xSemaphoreCreateBinary()` | 无需创建（任务自带） |
| ISR 给出 | `xSemaphoreGiveFromISR(s, &w)` | `vTaskNotifyGiveFromISR(h, &w)` |
| 任务等待 | `xSemaphoreTake(s, portMAX_DELAY)` | `ulTaskNotifyTake(pdTRUE, portMAX_DELAY)` |

### 超时处理

```c
uint32_t notify_count = ulTaskNotifyTake(pdTRUE, pdMS_TO_TICKS(100));

if (notify_count == 0) {
    printf("100ms 内没有收到通知\n");
} else {
    printf("收到 %lu 个通知（xClearCountOnExit=pdTRUE 时始终为 1）\n", notify_count);
}
```

> `ulTaskNotifyTake` 返回的是**清零前的通知值**。在 `pdTRUE` 模式下，通知值被清零前是 1 以上（每次 Give 增 1）。

## 模式二：带值通知（Notify / Wait）

### `xTaskNotify()` — 发送通知

```c
BaseType_t xTaskNotify(
    TaskHandle_t xTaskToNotify,
    uint32_t     ulValue,     // 要传递的值
    eNotifyAction eAction     // 如何更新通知值
);
```

**`eNotifyAction` 动作类型：**

| 动作 | 效果 | 适用场景 |
|------|------|---------|
| `eNoAction` | 只唤醒，不修改通知值 | 单纯的事件通知 |
| `eSetBits` | 通知值 = 原值 OR 新值（位掩码） | 累积状态标志 |
| `eIncrement` | 通知值 +1 | 等效 `xTaskNotifyGive` |
| `eSetValueWithOverwrite` | 直接覆盖旧值（总是成功） | 传递最新状态 |
| `eSetValueWithoutOverwrite` | 旧值未被消费则覆盖失败 | 防止数据丢失 |

### `xTaskNotifyWait()` — 等待通知

```c
BaseType_t xTaskNotifyWait(
    uint32_t  ulBitsToClearOnEntry,  // 进入等待前，清除通知值中的指定位
    uint32_t  ulBitsToClearOnExit,   // 退出等待时，清除通知值中的指定位
    uint32_t *pulNotificationValue,  // 出参：通知值
    TickType_t xTicksToWait          // 超时
);
```

- **返回：** `pdPASS`（收到通知）或 `pdFALSE`（超时）
- 通知值通过 `pulNotificationValue` 返回（可为 NULL）

### 位掩码模式示例

```c
#define EVT_TEMPERATURE_READY  (1 << 0)
#define EVT_HUMIDITY_READY     (1 << 1)
#define EVT_ERROR              (1 << 7)

/* 接收任务 */
void process_task(void *pv)
{
    uint32_t notification_value;

    while (1) {
        /* 等待通知，进入前不清零，退出时全清零 */
        if (xTaskNotifyWait(0, 0xFFFFFFFF, &notification_value,
                            portMAX_DELAY) == pdPASS) {

            if (notification_value & EVT_TEMPERATURE_READY) {
                printf("温度就绪\n");
            }
            if (notification_value & EVT_HUMIDITY_READY) {
                printf("湿度就绪\n");
            }
            if (notification_value & EVT_ERROR) {
                printf("传感器错误\n");
            }
        }
    }
}

/* ISR 中置位 */
void IRAM_ATTR temp_isr(void *arg)
{
    BaseType_t xHigherPriorityTaskWoken = pdFALSE;
    xTaskNotifyFromISR(xProcessTask, EVT_TEMPERATURE_READY,
                       eSetBits, &xHigherPriorityTaskWoken);
    portYIELD_FROM_ISR(xHigherPriorityTaskWoken);
}
```

### 带值覆盖模式（传递最新数据）

```c
/* 传感器采集 → 显示任务 */
void sensor_task(void *pv)
{
    while (1) {
        int16_t temp = ds18b20_get_temperature();

        /* 将温度值直接作为通知值发送 */
        xTaskNotify(xDisplayTask, (uint32_t)temp,
                    eSetValueWithOverwrite);
        vTaskDelay(pdMS_TO_TICKS(1000));
    }
}

void display_task(void *pv)
{
    uint32_t value;
    while (1) {
        xTaskNotifyWait(0, 0xFFFFFFFF, &value, portMAX_DELAY);
        printf("最新温度: %d.%d°C\n", (int16_t)value / 10,
               (int16_t)value % 10);
    }
}
```

## 在项目中的潜在应用

### 替代按键 ISR 的信号量

当前项目的按键使用**轮询模式**（`KeyInd_Scan` 每 10ms 读 GPIO），但未来如果改为中断驱动，任务通知是比信号量更好的选择：

```c
/* key.c — 任务通知替代信号量 */

TaskHandle_t xKeyTaskHandle = NULL;

void key_task(void *pvParameters)
{
    xKeyTaskHandle = xTaskGetCurrentTaskHandle();
    esp_task_wdt_add(NULL);

    while (1) {
        /* 阻塞等待按键中断（不占 CPU） */
        ulTaskNotifyTake(pdTRUE, pdMS_TO_TICKS(2000));

        KeyInd_Scan();
        KeyInd_HandleEvents();

        esp_task_wdt_reset();
    }
}

void IRAM_ATTR key_isr_handler(void *arg)
{
    BaseType_t xHigherPriorityTaskWoken = pdFALSE;
    vTaskNotifyGiveFromISR(xKeyTaskHandle, &xHigherPriorityTaskWoken);
    portYIELD_FROM_ISR(xHigherPriorityTaskWoken);
}
```

**优势：** 相比 `xSemaphoreGiveFromISR` / `xSemaphoreTake`：
- 无需创建信号量（省 RAM）
- 速度更快（约 30%）
- API 更简洁

### 替代单值队列

如果只需要在任务间传递**一个 32 位整数**（比如最新的传感器值），任务通知比队列更轻量：

```c
/* ❌ 队列方式（开销大） */
QueueHandle_t xQueue = xQueueCreate(1, sizeof(uint32_t));
uint32_t val = read_sensor();
xQueueSend(xQueue, &val, 0);

/* ✅ 任务通知方式（零开销） */
uint32_t val = read_sensor();
xTaskNotify(xReceiverTask, val, eSetValueWithOverwrite);
```

## 任务通知 vs 其他 IPC

| 对比维度 | 任务通知 | 二值信号量 | 队列 |
|---------|---------|-----------|------|
| **创建开销** | 零（内置） | 创建句柄 | 创建句柄 + 缓冲区 |
| **传递数据** | 32 位值 | 无 | 任意大小结构体 |
| **速度** | 最快 | 约 1.5× 通知时间 | 约 2× 通知时间 |
| **多消费者** | ❌ 不支持 | ✅ 多个任务可 Take | ✅ 但每条数据只给一个 |
| **广播** | ❌ | ❌ | ❌ |
| **通知累积** | ✅ 计数器累加 | ✅ 计数信号量 | ✅ 队列缓冲 |
| **ISR 安全** | ✅ | ✅ | ✅ |
| **超时等待** | ✅ | ✅ | ✅ |
| **内存占用** | 0 | 4~8 bytes | uxLength × uxItemSize |

## 注意事项

### 1. 通知值溢出

`ulTaskNotifyTake(pdFALSE, ...)` 模式下，每次 Give 通知值 +1。最大值 `~0UL`（0xFFFFFFFF），超过后回绕到 0。

```c
/* 计数器模式下可能溢出 */
for (int i = 0; i < 0x100000000; i++) {
    xTaskNotifyGive(xTask);  // 通知值最终会回绕
}
```

### 2. 多任务不能同时等待同一个通知

任务通知是点到点的——每个任务只有一个通知值。如果两个任务同时 `xTaskNotifyWait` 等待同一个发送者的通知，只会有一个被唤醒。

```c
/* ❌ 错误设计：两个任务等同一个通知 */
void task_a(void *pv) { xTaskNotifyWait(0,0,&val, portMAX_DELAY); }
void task_b(void *pv) { xTaskNotifyWait(0,0,&val, portMAX_DELAY); }
// 只有 task_a 会收到通知，task_b 永远阻塞

/* ✅ 正确：用队列或事件组实现多消费者 */
```

### 3. ISR 中必须用 FromISR 版本

```c
/* ❌ 错误 */
void IRAM_ATTR isr(void *arg) {
    xTaskNotify(xTask, val, eSetBits);  // ISR 中非法
}

/* ✅ 正确 */
void IRAM_ATTR isr(void *arg) {
    BaseType_t xHigherPriorityTaskWoken = pdFALSE;
    xTaskNotifyFromISR(xTask, val, eSetBits, &xHigherPriorityTaskWoken);
    portYIELD_FROM_ISR(xHigherPriorityTaskWoken);
}
```

### 4. 通知值使用前清零

如果复用通知值，确保在进入等待循环前清零：

```c
/* 创建任务后，通知值初始为 0。但如果之前用过，显式清零 */
ulTaskNotifyValueClear(xTask, 0xFFFFFFFF);
```

## 总结

| 要点 | 说明 |
|------|------|
| **零开销** | 每个任务内置通知值，无需创建内核对象 |
| **最快 IPC** | 比信号量快约 30%，比队列快约 50% |
| **点对点** | 只能一个任务通知另一个任务 |
| **32 位值** | 可传数值、位掩码、状态标志 |
| **两种模式** | Give/Take（信号量式）和 Notify/Wait（带值式） |
| **ISR 安全** | 用 `FromISR` 版本 + `portYIELD_FROM_ISR` |
| **替代场景** | 替代二值信号量、替代单值队列、替代事件组（简单 AND 场景） |
