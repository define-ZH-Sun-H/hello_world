# FreeRTOS 调试辅助使用指南

## 概述

FreeRTOS 提供了一系列调试工具，帮助开发者**检测栈溢出**、**分析 CPU 使用率**、**监控任务状态** 以及 **追踪内存分配**。在 ESP-IDF 环境下，还集成了 TWDT（任务看门狗）和堆内存调试功能。

---

## API 总览

```c
#include "freertos/FreeRTOS.h"
#include "freertos/task.h"

/* 栈检测 */
UBaseType_t uxTaskGetStackHighWaterMark(TaskHandle_t xTask);
UBaseType_t uxTaskGetStackHighWaterMark2(TaskHandle_t xTask);  // 32 位版本

/* 任务列表 */
void vTaskList(char *pcWriteBuffer);                 // 任务状态快照
void vTaskGetRunTimeStats(char *pcWriteBuffer);      // CPU 使用率统计

/* 运行时信息 */
void vTaskGetInfo(TaskHandle_t xTask, TaskStatus_t *pxTaskStatus,
                  BaseType_t xGetFreeStackSpace, eTaskState eState);
UBaseType_t uxTaskGetNumberOfTasks(void);
```

```c
#include "esp_task_wdt.h"

/* TWDT */
esp_err_t esp_task_wdt_add(TaskHandle_t task);
esp_err_t esp_task_wdt_reset(void);
```

```c
#include "esp_heap_caps.h"

/* 堆内存 */
size_t heap_caps_get_free_size(uint32_t caps);
size_t heap_caps_get_total_size(uint32_t caps);
void   heap_caps_print_heap_info(uint32_t caps);
```

## 栈高水位检测

### `uxTaskGetStackHighWaterMark()`

```c
UBaseType_t uxTaskGetStackHighWaterMark(TaskHandle_t xTask);
```

返回任务运行以来**栈剩余的最小字节数**。值越小，离栈溢出越近。

- 传 `NULL` 查自己
- **ESP-IDF 单位是 bytes**（原生 FreeRTOS 是 words）
- 应**在任务稳定运行一段时间后**调用（初始阶段栈使用通常较小）

### 使用

```c
void app_main(void)
{
    // ... 创建任务并稳定运行 ...

    vTaskDelay(pdMS_TO_TICKS(5000));  // 等任务运行一段时间

    printf("=== 栈使用情况 ===\n");
    printf("keytask:   %u bytes\n",
           uxTaskGetStackHighWaterMark(xTaskGetHandle("keytask")));
    printf("sensor:    %u bytes\n",
           uxTaskGetStackHighWaterMark(xTaskGetHandle("sensor_task")));
    printf("oled_disp: %u bytes\n",
           uxTaskGetStackHighWaterMark(xTaskGetHandle("oled_disp")));
}
```

### 评估标准

| 高水位值 | 等级 | 操作 |
|---------|------|------|
| > 500 bytes | ✅ 安全 | 可尝试减小栈大小以节省内存 |
| 200~500 bytes | ⚠️ 注意 | 当前安全，扩展功能后需重新评估 |
| 50~200 bytes | 🔴 危险 | 有溢出风险，应增加栈大小 |
| < 50 bytes | 💥 临界 | 栈随时可能溢出 |
| 0 | ❌ 已溢出 | 任务已踩到栈边界，可能已产生崩溃 |

### 栈调优示例

```c
/* 假设 uxTaskGetStackHighWaterMark("keytask") = 860 */
/* 初始栈 2048，剩余 860 → 实际使用约 1200 */
/* 
   如果剩余 >500，可以尝试将栈从 2048 减到 1536：

   xTaskCreatePinnedToCore(key_task, "keytask", 1536, NULL, 9, NULL, 0);

   省出 512 bytes 给其他任务用。
*/
```

**不要一次性减太多。** 每次减少不超过 25%，然后测试几小时确保没有栈溢出。

### 常见的栈溢出原因

```c
/* 1. 在栈上声明大数组 */
void my_task(void *pv)
{
    char buffer[1024];        // 占 1KB 栈！
    uint8_t framebuf[512];    // 又占 512 bytes！
    while (1) { ... }
}

/* 改进：挪到堆上 */
void my_task(void *pv)
{
    char *buffer = malloc(1024);
    uint8_t *framebuf = malloc(512);
    while (1) { ... }
    free(buffer);
    free(framebuf);
}

/* 2. 深层函数调用嵌套（以 printf 为代表的深层调用链） */
void level3() { printf("deep\n"); }  // printf 内部调用链很深
void level2() { level3(); }
void level1() { level2(); }
// 每个函数调用 + 局部变量都占用栈空间

/* 3. 递归 */
uint32_t factorial(uint32_t n) {
    return n <= 1 ? 1 : n * factorial(n - 1);  // 递归深度 = 栈消耗
}
```

## 任务列表

### `vTaskList()`

```c
void vTaskList(char *pcWriteBuffer);
```

输出所有任务的快照表格：

```
名称          状态  优先级  栈剩余  任务号
keytask        R      9       412     1
oled_disp      B      8      1024     2
sensor_task    B     10       876     3
IDLE           R      0      2048     4
Tmr Svc        B      1       512     5
```

| 状态 | 含义 |
|------|------|
| `R` | 运行态 或 就绪态 |
| `B` | 阻塞态 |
| `S` | 挂起态 |
| `D` | 已删除（资源待回收） |

**前置条件：** 需要使能以下配置：

```
idf.py menuconfig → FreeRTOS
  [*] Enable FreeRTOS trace facility  (configUSE_TRACE_FACILITY)
  [*] Enable FreeRTOS stats formatting functions  (configUSE_STATS_FORMATTING_FUNCTIONS)
```

**缓冲区大小：** 建议至少 1024 bytes，任务数多时可能需要 2048+：

```c
void print_task_list(void)
{
    char buf[2048];
    vTaskList(buf);
    printf("=== 任务列表 ===\n%s\n", buf);
}
```

## 运行时间统计

### `vTaskGetRunTimeStats()`

```c
void vTaskGetRunTimeStats(char *pcWriteBuffer);
```

输出各任务占用 CPU 的时间百分比：

```
名称         CPU%    时间（us）
keytask      12.3%   123456
oled_disp     5.1%    51234
sensor_task  18.7%   187654
IDLE         63.9%   640000
```

**用途：**
- 发现 CPU 占用异常高的任务
- 确认空闲任务有足够的余量
- 评估优先级设计是否合理

### 在 ESP-IDF 中启用

需要配置硬件计时器作为统计时基：

```
idf.py menuconfig → FreeRTOS
  [*] Enable FreeRTOS run time stats
      Choose time source for run time stats → Use ESP timer (CONFIG_FREERTOS_RUN_TIME_STATS_USING_ESP_TIMER)
```

或手动在 `sdkconfig` 中设置：

```
CONFIG_FREERTOS_GENERATE_RUN_TIME_STATS=y
CONFIG_FREERTOS_RUN_TIME_STATS_USING_ESP_TIMER=y
```

### 使用

```c
void print_cpu_usage(void)
{
    char buf[2048];
    vTaskGetRunTimeStats(buf);
    printf("=== CPU 使用率 ===\n%s\n", buf);
}
```

### 解读统计结果

| 空闲任务占比 | 结论 |
|------------|------|
| > 70% | ✅ CPU 余量充足 |
| 40~70% | ⚠️ 正常负载 |
| 10~40% | 🔴 CPU 繁忙 |
| < 10% | 💥 CPU 即将饱和，可能出现任务饿死 |

如果 IDLE 占比过低，考虑：
1. 降低任务的轮询频率（增大 `vTaskDelay`）
2. 将非实时任务移到另一个核心
3. 优化计算密集型任务的算法

## TWDT 调试

### TWDT 配置与初始化

```c
void wdt_initial(void)
{
    esp_task_wdt_deinit();  // bootloader 可能已初始化

    esp_task_wdt_config_t twdt_config = {
        .timeout_ms = 10000,                   // 10 秒超时
        .idle_core_mask = (1 << 0) | (1 << 1), // 两个核的空闲任务都喂狗
        .trigger_panic = true,                 // 超时触发 panic
    };
    esp_task_wdt_init(&twdt_config);
}
```

### 在任务中使用

```c
void key_task(void *pvParameters)
{
    esp_task_wdt_add(NULL);        // 当前任务加入 TWDT
    while (1) {
        // 工作 ...
        esp_task_wdt_reset();      // 手动喂狗
        vTaskDelay(pdMS_TO_TICKS(10));  // 阻塞自动喂狗
    }
}
```

### TWDT 常见错误日志

| 日志 | 含义 | 排查方向 |
|------|------|---------|
| `Task xxx got watchdog timeout` | xxx 任务超过 10s 未喂狗 | 检查该任务是否卡在死循环 |
| `CPU0: task_wdt timeout` | 核 0 空闲任务被饿死 | 核 0 上有高优先级任务未阻塞 |
| `CPU1: task_wdt timeout` | 核 1 空闲任务被饿死 | 核 1 上有高优先级任务未阻塞 |

### 哪些任务需要加入 TWDT

| 任务特征 | 建议 | 原因 |
|---------|------|------|
| 高优先级 (>5) | ✅ 必须加入 | 死循环会饿死空闲任务 |
| 含无限循环但无阻塞 | ✅ 必须加入 | 不喂狗就超时 |
| 定期 `vTaskDelay` | ✅ 建议加入 | 阻塞自动喂狗，开销小 |
| 低优先级 (<3) | ⚠️ 视情况 | 不太会饿死别人，但加入无害 |

## 堆内存调试

### `heap_caps_*` API

```c
#include "esp_heap_caps.h"

/* 查询各区域内存 */
size_t heap_caps_get_free_size(uint32_t caps);       // 空闲大小
size_t heap_caps_get_total_size(uint32_t caps);      // 总大小
void   heap_caps_print_heap_info(uint32_t caps);     // 打印详细

/* 从指定区域分配 */
void *heap_caps_malloc(size_t size, uint32_t caps);
```

**常用 `caps` 参数：**

| caps | 内存区域 | 典型用途 |
|------|---------|---------|
| `MALLOC_CAP_DMA` | 内部 DRAM（物理连续） | DMA 传输缓冲区 |
| `MALLOC_CAP_SPIRAM` | PSRAM（8MB） | 大缓冲区、帧缓冲 |
| `MALLOC_CAP_INTERNAL` | 内部 SRAM | 默认分配 |
| `MALLOC_CAP_8BIT` | 可字节寻址 | 普通数据 |
| `MALLOC_CAP_DEFAULT` | 默认（DRAM 优先） | 一般用途 |

### 内存使用查询

```c
void print_heap_info(void)
{
    printf("=== 堆内存 ===\n");
    printf("内部 DRAM 空闲: %u bytes\n",
           heap_caps_get_free_size(MALLOC_CAP_INTERNAL));
    printf("内部 DRAM 总计: %u bytes\n",
           heap_caps_get_total_size(MALLOC_CAP_INTERNAL));
    printf("PSRAM 空闲: %u bytes\n",
           heap_caps_get_free_size(MALLOC_CAP_SPIRAM));
    printf("PSRAM 总计: %u bytes\n",
           heap_caps_get_total_size(MALLOC_CAP_SPIRAM));

    /* 打印详细分配情况 */
    heap_caps_print_heap_info(MALLOC_CAP_DEFAULT);
}
```

### 内存泄漏排查

```c
/* 在可疑代码段前后打印空闲内存，观察差值 */
void test_memory_leak(void)
{
    size_t before = heap_caps_get_free_size(MALLOC_CAP_INTERNAL);

    // 执行可疑操作（如重复创建/删除任务）
    for (int i = 0; i < 100; i++) {
        TaskHandle_t task;
        xTaskCreate(test_task, "test", 2048, NULL, 5, &task);
        vTaskDelay(pdMS_TO_TICKS(10));
        vTaskDelete(task);
    }

    size_t after = heap_caps_get_free_size(MALLOC_CAP_INTERNAL);
    printf("内存变化: %d bytes (before=%u, after=%u)\n",
           (int)(after - before), before, after);
    // 如果 after < before 且差值不断增大 → 内存泄漏
}
```

### ESP-IDF 堆调试配置

```
idf.py menuconfig → Component config → Heap Memory Debugging
  [*] Enable heap memory debugging
  [*] Heap corruption checking
  [*] Heap poisoning (light)
  [*] Enable heap tracing
```

启用后 `heap_caps_print_heap_info()` 会输出更详细的信息。

## 在项目中集成调试

### 调试输出框架

本项目已有的分级调试宏：

```c
#include "debug.h"

DBG_ERROR("错误");     // 始终输出（红色）
DBG_WARN("警告");      // 默认输出
DBG_INFO("信息");      // 默认输出
DBG_DEBUG("调试");     // 仅在 DBG_LEVEL_DEBUG 时输出
```

在 `debug.h` 中控制输出级别：

```c
#define DBG_LEVEL_INFO      // 当前级别
// #define DBG_LEVEL_DEBUG   // 打开后可见 DBG_DEBUG 输出
// #define DBG_LEVEL_WARN    // 只显示警告和错误
// #define DBG_LEVEL_ERROR   // 只显示错误
```

### 调试命令处理器

添加一个串口命令，随时查看系统状态：

```c
void debug_command_handler(char cmd)
{
    char buf[2048];

    switch (cmd) {
    case 't':                         // 输入 't' → 任务列表
        vTaskList(buf);
        printf("=== 任务列表 ===\n%s\n", buf);
        break;

    case 'c':                         // 输入 'c' → CPU 使用率
        vTaskGetRunTimeStats(buf);
        printf("=== CPU 使用率 ===\n%s\n", buf);
        break;

    case 's':                         // 输入 's' → 栈使用
        printf("=== 栈使用 ===\n");
        printf("keytask:   %u\n",
               uxTaskGetStackHighWaterMark(xTaskGetHandle("keytask")));
        printf("sensor:    %u\n",
               uxTaskGetStackHighWaterMark(xTaskGetHandle("sensor_task")));
        printf("oled_disp: %u\n",
               uxTaskGetStackHighWaterMark(xTaskGetHandle("oled_disp")));
        break;

    case 'm':                         // 输入 'm' → 堆内存
        print_heap_info();
        break;

    default:
        printf("未知命令: %c\n", cmd);
    }
}
```


## 常见问题诊断

### 问题 1：栈溢出

**症状：** 系统随机崩溃、`Guru Meditation Error: Core 1 panic'ed`、函数调用返回地址异常

**排查步骤：**

```
1. 启用栈溢出检测
   idf.py menuconfig → FreeRTOS → [*] Check for stack overflow
   选项: "Check by stack canary" 或 "Check by stack pointer"

2. 在 app_main 中打印所有任务的高水位

3. 如果高水位接近 0 → 增加对应任务的栈大小

4. 检查是否有大数组在栈上声明（改为 malloc / heap_caps_malloc）

5. 检查递归调用深度
```

### 问题 2：TWDT 超时复位

**症状：** 日志 `Task xxx got watchdog timeout`，芯片反复复位

**排查步骤：**

```
1. 确认该任务调用了 esp_task_wdt_add(NULL)

2. 检查主循环是否有阻塞操作
   - vTaskDelay / xQueueReceive / xSemaphoreTake 会自动喂狗
   - 纯计算循环不会喂狗，需要手动 esp_task_wdt_reset()

3. 检查优先级是否过高导致低优先级任务无法运行
   - 如果任务优先级为 10 且不阻塞，核上所有 <10 的任务都得不到 CPU
```

### 问题 3：内存泄漏

**症状：** 运行一段时间后 `heap_caps_get_free_size` 逐渐减少，最终 malloc 返回 NULL

**排查步骤：**

```
1. 在可疑代码块前后打印空闲内存差值

2. 检查是否有创建后未删除的内核对象
   - xQueueCreate → vQueueDelete
   - xTaskCreate → vTaskDelete（任务用完要删）
   - xTimerCreate → xTimerDelete

3. 检查 malloc 是否有对应的 free
   - 特别是错误路径中容易遗漏 free

4. 启用堆追踪（heap tracing）定位泄漏点
```

### 问题 4：优先级反转

**症状：** 高优先级任务响应延迟、系统行为间歇性异常

**排查步骤：**

```
1. 检查所有共享资源的保护方式
   - xSemaphoreCreateBinary（二值信号量）→ 不支持优先级继承
   - xSemaphoreCreateMutex（互斥锁）→ 支持优先级继承

2. 检查低优先级任务持有锁时是否被中优先级任务抢占

3. 用 vTaskGetRunTimeStats 观察各任务 CPU 占比
```

## 总结

| 工具 | 用途 | 前置条件 |
|------|------|---------|
| `uxTaskGetStackHighWaterMark` | 栈溢出检测 | 无 |
| `vTaskList` | 查看所有任务状态 | `configUSE_TRACE_FACILITY` |
| `vTaskGetRunTimeStats` | CPU 使用率分析 | `configGENERATE_RUN_TIME_STATS` |
| `esp_task_wdt_add/reset` | 任务死锁检测 | `esp_task_wdt_init` |
| `heap_caps_get_free_size` | 堆内存监控 | 无 |
| `heap_caps_print_heap_info` | 堆分配详情 | 无 |
| `DBG_INFO/DEBUG/ERROR` | 分级日志输出 | `debug.h` |
