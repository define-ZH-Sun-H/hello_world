# FreeRTOS 任务管理使用指南

## 什么是 FreeRTOS 任务

任务是 FreeRTOS 中最基本的执行单元。每个任务是一个 **无限循环的 C 函数**，拥有独立的 **栈空间** 和 **优先级**。调度器根据优先级和时间片轮转决定哪个任务运行。

```
任务 A (优先级 5)  ═══╗══════════╗══════╗══════
任务 B (优先级 3)     ║  ═══╗═══║══╗═══║═════
任务 C (优先级 1)     ║  ║  ║  ║  ║  ║  ═══
空闲任务 (优先级 0)    ═══╝══╝══╝══╝══╝══╝════
```

### 任务状态

```
               ┌─────────────┐
               │    运行态    │ ← 实际占用 CPU
               └──────┬──────┘
                      │
    ┌─────────────────┼─────────────────┐
    │                 │                 │
    ▼                 ▼                 ▼
┌────────┐     ┌──────────┐     ┌──────────┐
│ 就绪态  │ ◄─► │  阻塞态   │     │  挂起态   │
└────────┘     └──────────┘     └──────────┘
                vTaskDelay()    vTaskSuspend()
                等待队列/信号量   vTaskResume()
```

| 状态 | 含义 | 如何进入 | 如何退出 |
|------|------|---------|---------|
| **运行态** | 正在占用 CPU | 调度器选择 | 被更高优先级任务抢占 / 主动阻塞 |
| **就绪态** | 可运行但 CPU 被占 | 阻塞条件解除 | 调度器选择运行 |
| **阻塞态** | 等待某事件 | `vTaskDelay`, `xQueueReceive` 等 | 延时到 / 数据到达 / 超时 |
| **挂起态** | 手动暂停 | `vTaskSuspend` | `vTaskResume` |

## API 总览

```c
#include "freertos/FreeRTOS.h"
#include "freertos/task.h"

/* 创建任务 */
BaseType_t xTaskCreatePinnedToCore(TaskFunction_t pvTaskCode,
    const char * const pcName, const uint32_t usStackDepth,
    void * const pvParameters, UBaseType_t uxPriority,
    TaskHandle_t * const pvCreatedTask, const BaseType_t xCoreID);

/* 任务控制 */
void vTaskDelay(TickType_t xTicksToDelay);              // 相对延时
void vTaskDelayUntil(TickType_t *pxLastWake, TickType_t xIncrement);  // 绝对延时
void vTaskSuspend(TaskHandle_t xTask);                  // 挂起
void vTaskResume(TaskHandle_t xTask);                   // 恢复
void vTaskDelete(TaskHandle_t xTask);                   // 删除

/* 查询 */
TickType_t   xTaskGetTickCount(void);                   // 当前 tick
TaskHandle_t xTaskGetCurrentTaskHandle(void);            // 自己的句柄
TaskHandle_t xTaskGetHandle(const char *pcName);        // 按名称查句柄
UBaseType_t  uxTaskGetStackHighWaterMark(TaskHandle_t); // 栈余量
```

## 任务创建

### `xTaskCreatePinnedToCore()`（ESP-IDF 扩展）

```c
BaseType_t xTaskCreatePinnedToCore(
    TaskFunction_t       pvTaskCode,      // 任务函数指针
    const char * const   pcName,          // 任务名（调试用）
    const uint32_t       usStackDepth,    // 栈大小，单位 bytes
    void * const         pvParameters,    // 传给任务函数的参数
    UBaseType_t          uxPriority,      // 优先级 0~24
    TaskHandle_t * const pvCreatedTask,   // 出参，任务句柄
    const BaseType_t     xCoreID          // 0=PRO_CPU, 1=APP_CPU
);
```

**参数详解：**

| 参数 | 说明 |
|------|------|
| `pvTaskCode` | 必须是一个永不返回的 `void func(void*)`，内部通常是 `while(1)` |
| `pcName` | 最大 `configMAX_TASK_NAME_LEN`（默认 16），用于调试和 `xTaskGetHandle()` |
| `usStackDepth` | **ESP-IDF 单位是 bytes**（原生 FreeRTOS 是 words=4 bytes） |
| `pvParameters` | 可在创建时传任意结构体指针，任务生命周期内有效 |
| `uxPriority` | 0~`configMAX_PRIORITIES`-1（默认 25），数字越大优先级越高 |
| `pvCreatedTask` | 可用于后续 `vTaskDelete`、`vTaskNotifyGiveFromISR` 等 |
| `xCoreID` | `0`=PRO_CPU, `1`=APP_CPU, `tskNO_AFFINITY`=调度器决定 |

**返回：** `pdPASS` 或 `errCOULD_NOT_ALLOCATE_REQUIRED_MEMORY`

### 与原生 FreeRTOS 的差异

ESP-IDF 的 FreeRTOS 与原生 FreeRTOS 有几个关键差异：

| 差异项 | 原生 FreeRTOS | ESP-IDF |
|--------|-------------|--------|
| 任务创建 | `xTaskCreate()` | `xTaskCreate()` + `xTaskCreatePinnedToCore()` |
| 栈单位 | words（4 bytes） | **bytes** |
| `vTaskStartScheduler()` | 必须手动调用 | **禁止调用**，IDF 自动启动 |
| 核绑定 | 无 | 必须指定 xCoreID |

### 任务函数的标准结构

```c
void my_task(void *pvParameters)
{
    /* 1. 初始化（只执行一次） */
    int param = (int)(intptr_t)pvParameters;
    esp_task_wdt_add(NULL);              // 需要时加入看门狗

    /* 2. 主循环 */
    while (1) {
        // 工作逻辑
        esp_task_wdt_reset();            // 喂狗
        vTaskDelay(pdMS_TO_TICKS(10));   // 阻塞让出 CPU
    }

    /* 3. 如果跳出循环，必须删除自己 */
    vTaskDelete(NULL);
}
```

## 优先级与调度

### ESP32-S3 双核调度规则

1. **每个核独立调度**：核 0 和核 1 各自运行当前就绪态中优先级最高的任务
2. **同优先级轮转**：相同优先级的任务按时间片（默认 10ms）轮转
3. **优先级 0 保留**给空闲任务
4. **看门狗风险**：高优先级任务如果死循环不阻塞，低优先级任务（含空闲任务）永远得不到运行

### 优先级设计原则

```c
/* 本项目中的任务优先级 */
#define PRIO_SENSOR   10   // 传感器采集 — 最高，数据实时性要求高
#define PRIO_KEY       9   // 按键扫描 — 响应速度要求高
#define PRIO_DISPLAY   8   // OLED 显示 — 可容忍轻微抖动
```

**优先级选择建议：**

| 任务类型 | 优先级 | 理由 |
|---------|-------|------|
| 硬实时（传感器、控制） | 9~10 | 必须按时响应 |
| 交互（按键、通信） | 6~8 | 用户可感知延迟但容忍度低 |
| UI 显示 | 4~6 | 帧率抖动不明显 |
| 后台维护 | 1~3 | 可被任何其他任务抢占 |

如果一个任务的优先级设为 `0`，它将与空闲任务共享 CPU 时间——空闲任务的优先级就是 0。

### 优先级反转问题

```
低优先级任务 L 持有锁
        ↓
高优先级任务 H 等待锁 → 被阻塞
        ↓
中优先级任务 M 运行（不涉及该锁）→ L 无法运行 → 锁无法释放 → H 一直等
```

解决办法：使用互斥锁（`xSemaphoreCreateMutex`，支持优先级继承）而非二值信号量保护共享资源。

## 任务延时

### `vTaskDelay()` — 相对延时

从**调用时刻**开始计时，经过指定 tick 数后回到就绪态。

```c
vTaskDelay(pdMS_TO_TICKS(100));   // 延时 100ms
vTaskDelay(pdMS_TO_TICKS(0));     // 让出 CPU（同优先级轮转）
```

**注意：** `pdMS_TO_TICKS(9)` 在 100Hz tick 下结果为 0，等于 `vTaskDelay(0)`，不会产生实际延时。

### `vTaskDelayUntil()` — 绝对延时（固定周期）

以初始时刻为基准，消除累积误差：

```c
/* ❌ 累积误差 */
while (1) {
    do_work();                          // 假设耗时 3ms
    vTaskDelay(pdMS_TO_TICKS(10));      // 实际周期 = 13ms, 10ms, 13ms...
}

/* ✅ 固定周期 */
TickType_t xLastWakeTime = xTaskGetTickCount();
while (1) {
    do_work();
    vTaskDelayUntil(&xLastWakeTime, pdMS_TO_TICKS(10));  // 始终 10ms
}
```

| 特性 | `vTaskDelay()` | `vTaskDelayUntil()` |
|------|---------------|--------------------|
| 计时基准 | 调用时刻 | 上一次唤醒时刻 |
| 误差 | 累积 | 固定 |
| 适用场景 | 一次延时、非周期等待 | 固定周期任务 |

## 栈管理

### 栈大小估算

ESP-IDF 中栈单位是 **bytes**，不同任务的建议初始值：

| 任务类型 | 建议栈大小 | 典型用途 |
|---------|-----------|---------|
| 简单按键扫描 | 2048 bytes | 少量局部变量，无深层调用 |
| 传感器 + printf | 2560 bytes | printf 内部缓冲区较大 |
| OLED 显示 | 3072 bytes | 可能含字体表操作、帧缓冲 |
| WiFi 网络 | 4096+ bytes | LWIP 协议栈调用栈较深 |

### 栈溢出检测

硬件栈监控 + 高水位检测双保险：

```c
/* 在 app_main 末尾或串口命令中调用 */
void print_stack_usage(void)
{
    printf("=== 栈使用情况 ===\n");
    printf("keytask:   %u bytes 剩余\n",
           uxTaskGetStackHighWaterMark(xTaskGetHandle("keytask")));
    printf("sensor:    %u bytes 剩余\n",
           uxTaskGetStackHighWaterMark(xTaskGetHandle("sensor_task")));
    printf("oled_disp: %u bytes 剩余\n",
           uxTaskGetStackHighWaterMark(xTaskGetHandle("oled_disp")));
}
```

| 高水位值 | 结论 | 操作 |
|---------|------|------|
| > 500 bytes | 安全 | 可尝试减小栈大小省内存 |
| 100~500 bytes | 注意 | 当前安全，但添加新功能后可能溢出 |
| < 100 bytes | **危险** | 立即增加栈大小 |
| 0 | **已溢出** | 任务已踩到栈边界，可能已产生奇怪 bug |

## 看门狗集成

### TWDT 配置

```c
void wdt_initial(void)
{
    esp_task_wdt_deinit();  // bootloader 可能已初始化

    esp_task_wdt_config_t twdt_config = {
        .timeout_ms = 10000,
        .idle_core_mask = (1 << 0) | (1 << 1),
        .trigger_panic = true,
    };
    esp_task_wdt_init(&twdt_config);
}
```

### 在任务中使用

```c
void key_task(void *pvParameters)
{
    esp_task_wdt_add(NULL);   // 加入监控
    while (1) {
        do_work();
        esp_task_wdt_reset(); // 显式喂狗
        vTaskDelay(pdMS_TO_TICKS(10)); // 阻塞自动喂狗
    }
}
```

| 操作 | 是否喂狗 |
|------|---------|
| `vTaskDelay()` | ✅ 自动喂狗（任务不在运行态） |
| `xQueueReceive()` 阻塞 | ✅ 自动喂狗 |
| `xSemaphoreTake()` 阻塞 | ✅ 自动喂狗 |
| 纯计算不阻塞 | ❌ 必须手动 `esp_task_wdt_reset()` |

## 核绑定策略

ESP32-S3 是双核芯片，合理的负载分配很重要：

```
PRO_CPU (core 0)          APP_CPU (core 1)
┌────────────────────┐   ┌────────────────────┐
│ WiFi / BLE 协议栈   │   │ 显示任务 (oled_disp)│
│ 按键任务 (keytask)   │   │ 用户 UI 处理       │
│ 传感器采集 (sensor)  │   │                    │
│ 中断处理 (ISR)      │   │                    │
└────────────────────┘   └────────────────────┘
```

**推荐分配原则：**

- **PRO_CPU（core 0）**：中断处理、I/O 密集、实时性要求高的任务
- **APP_CPU（core 1）**：UI 更新、非实时处理、重量级计算

**为什么？** ESP-IDF 的部分驱动（WiFi、BLE 协议栈）默认运行在 PRO_CPU，将 I/O 任务也放在 PRO_CPU 可以减少核间同步开销。

## 常见陷阱

### 1. 忘记 `vTaskDelay` 或阻塞

```c
/* ❌ 高优先级死循环 — 饿死所有低优先级任务 */
void high_prio_task(void *pv)
{
    while (1) {
        do_something();
        // 没有阻塞，没有喂狗 → TWDT 复位
    }
}

/* ✅ 加入阻塞 */
void high_prio_task(void *pv)
{
    while (1) {
        do_something();
        vTaskDelay(pdMS_TO_TICKS(10));  // 让出 CPU
    }
}
```

### 2. 任务未加入 TWDT

高优先级任务如果在 Phase 3（`create_all_tasks`）中创建但忘记加入 TWDT，一旦死循环不会自动复位，导致设备卡死。

### 3. 在 ISR 中创建/删除任务

`xTaskCreatePinnedToCore` 和 `vTaskDelete` 不能在中 ISR 中调用。如果需要在 ISR 中唤醒任务创建逻辑，用 `xTaskNotifyGiveFromISR` 通知一个管理任务来处理。

### 4. 栈大小误算

```c
/* 栈上声明大数组容易导致溢出 */
void my_task(void *pv)
{
    char big_buffer[2048];  // 一下就占了 2KB 栈！
    while (1) {
        // ...
    }
}

/* 改进：放堆里 */
void my_task(void *pv)
{
    char *big_buffer = malloc(2048);  // 或者 heap_caps_malloc
    while (1) {
        // ...
    }
    free(big_buffer);
}
```

## 总结

| 原则 | 说明 |
|------|------|
| **任务函数永不返回** | 用 `while(1)` 包裹主逻辑，跳出后调 `vTaskDelete(NULL)` |
| **栈单位是 bytes** | ESP-IDF 下不同于原生 FreeRTOS |
| **高优先级必须阻塞或喂狗** | 否则饿死低优先级任务 + 触发 TWDT |
| **临界资源用互斥锁** | 二值信号量没有优先级继承 |
| **固定周期用 DelayUntil** | 避免累积误差 |
| **统一创建任务** | 在 app_main Phase 3 集中管理，方便查看栈/优先级/核绑定 |
